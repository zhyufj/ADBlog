# 软件详细设计说明书

## 1. 概述

本文档描述 `main.go` 中每个函数的实现逻辑、核心算法和异常处理细节。

---

## 2. 函数详细设计

### 2.1 main() — 主流程控制

**功能：** 编排整体交互流程，依次调用各子模块。

**流程图：**

```
开始
  │
  ▼
selectDevice(reader)
  │
  ├─ device == "" ──▶ 输出 "未检测到设备" ──▶ 结束
  │
  ▼
输入包名
  │
  ├─ pkg == "9" ──▶ listPackages(device) ──▶ 再次输入包名
  ├─ pkg == "0" ──▶ getCurrentPackage(device)
  │                   ├─ 成功 ──▶ 输出 "检测到当前应用: <pkg>"
  │                   └─ 失败 ──▶ 输出提示，pkg = ""
  └─ 其他/空 ──▶ 直接使用
  │
  ▼
chooseLevel(reader) ──▶ level
  │
  ▼
输出配置信息（设备、级别、包名）
  │
  ▼
startLogcat(device, pkg, level)
  │
  ▼
结束
```

**异常处理：**
- `reader.ReadString('\n')` 读取失败时，pkg 置为空字符串，不中断流程。
- 无需用户输入即可完成的步骤（如 selectDevice 单设备场景）不阻塞。

### 2.2 selectDevice() — 设备检测与选择

**函数签名：** `func selectDevice(reader *bufio.Reader) string`

**输入参数：**
- `reader`: 标准输入读取器，用于多设备场景的用户选择

**返回值：**
- 设备序列号（成功）
- `""`（无可用设备或 ADB 执行失败）

**核心算法：**

```
1. exec.Command(adbPath, "devices").CombinedOutput()
2. 按行分割输出
3. 遍历每一行:
   a. TrimSpace 去除首尾空白
   b. 跳过空行和 "List of devices" 开头的行
   c. strings.Fields() 按空白分割
   d. 若 fields[1] == "device"，将 fields[0] 加入设备列表
4. 判断设备数量:
   - 0 台 → 输出 ADB 原始输出 → return ""
   - 1 台 → 输出设备号 → return devices[0]
   - N 台 → 列出设备列表 → 读取用户选择
     - 有效选择 (1~N) → return devices[choice-1]
     - 无效选择 → 输出提示 → return devices[0]（默认第一台）
```

**校验规则：**
- 仅接受 `fields[1] == "device"` 的设备
- `unauthorized` / `offline` 状态的设备自动排除
- 用户输入非数字或越界时，容错为默认第一台

**输出示例：**

```
# 单设备
检测到设备: ABC123456789

# 多设备
检测到多个设备:
1. ABC123456789
2. DEF987654321
请选择设备: 2

# 无设备
ADB输出:
List of devices attached

未检测到设备
```

### 2.3 getCurrentPackage() — 获取当前前台应用包名

**函数签名：** `func getCurrentPackage(device string) string`

**输入参数：**
- `device`: 设备序列号

**返回值：**
- 包名字符串（成功）
- `""`（无法获取时）

**核心算法：**

```
1. exec.Command(adbPath, "-s", device, "shell", "dumpsys", "window").Output()
2. 按行分割输出
3. 遍历每一行:
   a. strings.Contains(line, "mCurrentFocus")
   b. 找到后，strings.Fields() 分割该行
   c. 遍历每个字段:
      - 若 strings.Contains(part, "/")
      - return strings.Split(part, "/")[0]  // 取 "/" 之前的部分
4. 未找到 → return ""
```

**数据示例：**

```
# 输入 (dumpsys window 输出)
WINDOW MANAGER POLICY STATE (dumpsys window policy)
  mCurrentFocus=Window{abc123 u0 com.example.app/com.example.app.MainActivity}
  mFocusedApp=AppWindowToken{def456 token=...}

# 解析过程
line: "  mCurrentFocus=Window{abc123 u0 com.example.app/com.example.app.MainActivity}"
parts: ["mCurrentFocus=Window{abc123", "u0", "com.example.app/com.example.app.MainActivity}"]
  → part "com.example.app/com.example.app.MainActivity}" contains "/"
  → split by "/": ["com.example.app", "com.example.app.MainActivity}"]
  → return "com.example.app"
```

**边界情况：**
- 锁屏状态：`mCurrentFocus=null` → 无 "/" 包含字段 → 返回 `""`
- 桌面状态：返回 Launcher 包名（如 `com.android.launcher3`）

### 2.4 listPackages() — 列出所有应用包名

**函数签名：** `func listPackages(device string)`

**输入参数：**
- `device`: 设备序列号

**输出：** 直接输出到 stdout，无返回值。

**核心算法：**

```
1. 输出 "正在获取所有APP包名..."
2. exec.Command(adbPath, "-s", device, "shell", "pm", "list", "packages").Output()
3. 按行分割输出
4. 遍历每一行:
   a. TrimSpace 去除首尾空白
   b. 若 strings.HasPrefix(line, "package:"):
      - strings.TrimPrefix(line, "package:") → 包名
      - fmt.Println(pkg)
```

**异常处理：**
- `cmd.Output()` 失败时输出错误信息并返回，不中断主流程。

### 2.5 chooseLevel() — 日志级别选择

**函数签名：** `func chooseLevel(reader *bufio.Reader) string`

**输入参数：**
- `reader`: 标准输入读取器

**返回值：**
- 级别字母：`"F"`, `"E"`, `"W"`, `"I"`, `"D"`, `"V"` 之一
- 默认值：`"E"`

**处理流程：**

```
1. 输出菜单:
   "1. F - 严重"
   "2. E - 错误"
   "3. W - 警告"
   "4. I - 信息"
   "5. D - 调试"
   "6. V - 全部"
   "选择 (默认2): "

2. 读取用户输入 → strings.TrimSpace(input)

3. switch 匹配:
   case "1" → return "F"
   case "2" → return "E"
   case "3" → return "W"
   case "4" → return "I"
   case "5" → return "D"
   case "6" → return "V"
   default  → return "E"    // 包括空输入、无效数字、非数字字符

4. 读取异常 → return "E"
```

**输入校验表：**

| 用户输入 | TrimSpace 后 | 匹配结果 |
|----------|-------------|----------|
| `1\n` | `"1"` | `"F"` |
| `2\n` | `"2"` | `"E"` |
| `3\n` | `"3"` | `"W"` |
| `4\n` | `"4"` | `"I"` |
| `5\n` | `"5"` | `"D"` |
| `6\n` | `"6"` | `"V"` |
| `\n` | `""` | default → `"E"` |
| `7\n` | `"7"` | default → `"E"` |
| `0\n` | `"0"` | default → `"E"` |
| `abc\n` | `"abc"` | default → `"E"` |
| `-1\n` | `"-1"` | default → `"E"` |
| ` 3 \n` | `"3"` | `"W"` (trim 后有效) |

### 2.6 startLogcat() — 启动日志抓取

**函数签名：** `func startLogcat(device, pkg, minLevel string)`

**输入参数：**
- `device`: 设备序列号
- `pkg`: 包名过滤关键字（空字符串表示不过滤）
- `minLevel`: 最低日志级别

**核心算法：**

```
1. 清空缓冲区:
   exec.Command(adbPath, "-s", device, "logcat", "-c").Run()
   (忽略错误，继续执行)

2. 启动 logcat:
   cmd := exec.Command(adbPath, "-s", device, "logcat")
   cmd.Stderr = cmd.Stdout          // stderr 合并到 stdout
   stdout, _ := cmd.StdoutPipe()    // 创建管道
   cmd.Start()

3. 逐行读取:
   scanner := bufio.NewScanner(stdout)
   buf := make([]byte, 1024*1024)   // 1MB 缓冲区
   scanner.Buffer(buf, bufio.MaxScanTokenSize)

   for scanner.Scan() {
       line := scanner.Text()
       processLine(line, pkg, minLevel)
   }

4. 处理异常:
   scanner.Err() → 输出 "日志读取错误"
   cmd.Wait() → 等待进程结束
```

**缓冲区设计：**
- 默认 `bufio.Scan` 单行最大 64KB，不能满足超长日志需求
- 自定义缓冲区设为 1MB，兼容堆栈信息等超长行

### 2.7 processLine() — 单行日志过滤与分发

**函数签名：** `func processLine(line, pkg, minLevel string)`

**输入参数：**
- `line`: 单行原始日志内容
- `pkg`: 包名过滤关键字
- `minLevel`: 最低日志级别

**三阶段过滤逻辑：**

```
┌─────────────────────────────────┐
│ 阶段 1：包名过滤                  │
│ if pkg != "" && !Contains(line, pkg) │
│     return                      │
└──────────────┬──────────────────┘
               │ 通过
               ▼
┌─────────────────────────────────┐
│ 阶段 2：级别提取                  │
│ level := extractLevel(line)     │
│ if level == ""                  │
│     return                      │
└──────────────┬──────────────────┘
               │ 通过
               ▼
┌─────────────────────────────────┐
│ 阶段 3：级别阈值过滤              │
│ if levelOrder[level] < levelOrder[minLevel] │
│     return                      │
└──────────────┬──────────────────┘
               │ 通过
               ▼
┌─────────────────────────────────┐
│ printColored(line, level)       │
└─────────────────────────────────┘
```

**过滤条件真值表（示例）：**

| 包名匹配 | 级别识别 | 级别≥阈值 | 结果 |
|----------|----------|----------|------|
| ✓ | ✓ | ✓ | 输出（带颜色） |
| ✓ | ✓ | ✗ | 过滤 |
| ✓ | ✗ | — | 过滤 |
| ✗ | — | — | 过滤 |
| 不过滤包名 | ✓ | ✓ | 输出 |
| 不过滤包名 | ✓ | ✗ | 过滤 |

### 2.8 extractLevel() — 日志级别提取（核心算法）

**函数签名：** `func extractLevel(line string) string`

**设计背景：** 旧版本使用 `strings.Contains` 在整行中搜索 " X/" 或 " X "，存在严重缺陷——若日志消息体（message body）中包含 "I/Stream"、"D/Config" 等文本，会被错误识别为日志级别标记，导致：
- 过滤判断错误（例如 E 级日志被误判为 I 级，可能被过滤器错误保留或丢弃）
- 颜色输出错误

**新版算法：逐字符精确定位匹配**

```
输入: line (原始日志行)
输出: 级别字母 "V"/"D"/"I"/"W"/"E"/"F" 或 ""

算法步骤:

1. 确定检查范围:
   checkRange := line
   if len(line) > 50:
       checkRange = line[:50]

2. 遍历 checkRange 的每个字符 (索引 i):
   for i, ch := range checkRange:
       
       a. 字符过滤:
          if ch ∉ {'V','D','I','W','E','F'}:
              continue
       
       b. 前导字符验证:
          if i > 0 && checkRange[i-1] != ' ':
              continue
          // 前导为行首(i==0)或空格(i>0且前一个字符为空格)
       
       c. 后继字符验证:
          if i+1 < len(checkRange):
              next := checkRange[i+1]
              if next == '/' || next == ' ':
                  return string(ch)   // 匹配成功
              // else: 继续遍历 (后继不是 / 或空格)

3. 遍历结束未匹配 → return ""
```

**匹配规则总结：**

| 位置条件 | 示例 | 匹配？ |
|----------|------|--------|
| 行首 + 字母 + `/` | `E/TAG:` | ✓ |
| 行首 + 字母 + 空格 | `E TAG:` | ✓ |
| 空格 + 字母 + `/` | ` I/TAG:` | ✓ |
| 空格 + 字母 + 空格 | ` I TAG:` | ✓ |
| 字母在单词中 + `/` | `FIRE/` | ✗（前导非空格） |
| 等号 + 字母 + `/` | `=D/test` | ✗（前导非空格） |
| 空格 + 字母 + 其他 | ` F-TAG` | ✗（后随非/非空格） |

**为什么只检查前 50 字符？**
- Android logcat 的级别标记总是出现在日志行的前部（时间戳之后、Tag 之前）
- 限制检查范围避免了消息体中的偶然匹配
- 50 字符足以覆盖所有标准 logcat 输出格式

**旧版 vs 新版对比：**

```
日志行: "E/MyApp(12345): Error reading I/Stream data"

旧版 (strings.Contains):
  ├─ Contains(prefix, " I/") → true → 返回 "I"  ❌ 误判！
  └─ 实际级别: E

新版 (逐字符扫描):
  ├─ i=0, ch='E', 前导=行首 ✓, 后继='/' ✓ → 返回 "E"  ✓ 正确！
  └─ i=33, ch='I', 前导=' ' ✓, 后继='/' ✓ — 但循环已在 i=0 返回
```

### 2.9 printColored() — 彩色输出

**函数签名：** `func printColored(line, level string)`

**输入参数：**
- `line`: 要输出的日志文本
- `level`: 日志级别字母

**颜色映射表：**

| level | ANSI 转义码 | 效果 | 语义 |
|-------|-------------|------|------|
| `"F"` | `\033[41;37m` + line + `\033[0m` | 红底白字 | Fatal，需要立即关注 |
| `"E"` | `\033[31m` + line + `\033[0m` | 红色文字 | Error，错误信息 |
| `"W"` | `\033[33m` + line + `\033[0m` | 黄色文字 | Warning，警告信息 |
| `"I"` | `\033[32m` + line + `\033[0m` | 绿色文字 | Info，常规信息 |
| `"D"` | `\033[36m` + line + `\033[0m` | 青色文字 | Debug，调试信息 |
| 其他 | line（原始输出） | 终端默认色 | Verbose 及未知级别 |

**ANSI 转义码结构：**
- `\033[` — CSI (Control Sequence Initiator)
- `41` — 背景色为红色
- `37` — 前景色为白色
- `m` — SGR (Select Graphic Rendition) 终止符
- `\033[0m` — 重置所有属性

## 3. 全局配置设计

### 3.1 adbPath

```go
var adbPath = ".\\adb.exe"
```

- Windows 相对路径格式
- 要求 adb.exe 与程序在同一目录
- 使用 `exec.Command(adbPath, ...)` 统一调用

### 3.2 levelOrder

```go
var levelOrder = map[string]int{
    "V": 0, "D": 1, "I": 2, "W": 3, "E": 4, "F": 5,
}
```

- 键：日志级别字母
- 值：级别权值（越大越严重）
- 用途：`levelOrder[日志级别] < levelOrder[最低级别]` 决定是否过滤

**设计原则：** 权值差值无实际意义，仅用于大小比较。

## 4. 错误处理策略

| 错误场景 | 处理方式 |
|----------|----------|
| adb devices 执行失败 | 输出 ADB 错误信息 + "未检测到设备"，程序退出 |
| adb shell dumpsys 失败 | 返回空包名，提示用户并继续（不中断） |
| adb shell pm list 失败 | 输出错误信息，继续包名输入流程 |
| logcat -c 失败 | 忽略错误（缓冲区清空失败不影响日志查看） |
| logcat 启动失败 | 输出 "logcat启动失败"，函数返回 |
| 日志读取错误 | 输出 "日志读取错误"，结束日志流 |
| 用户输入读取异常 | 使用默认值（设备=第一台，包名=空，级别=E） |
| 设备序列号含特殊字符 | Go 的 `exec.Command` 参数自动转义 |

## 5. 关键 Bug 修复记录

### Bug #1：消息体中的级别文本导致误判（已修复）

| 项目 | 内容 |
|------|------|
| **严重程度** | P0 — 导致日志过滤结果错误 |
| **现象** | E 级别日志 `E/MyApp: Error reading I/Stream data` 被误判为 I 级别 |
| **根因** | `extractLevel` 使用 `strings.Contains` 在整行中搜索，消息体中的 ` I/` 匹配了 I 级别 |
| **修复方案** | 改为逐字符扫描 + 前导/后继验证，确保只匹配位置正确的级别标记 |
| **验证** | 新增 `TestExtractLevel_MessageBodyFalseMatch` 6 条单元测试 |

### Bug #2：多设备选择无效时无提示（已修复）

| 项目 | 内容 |
|------|------|
| **严重程度** | P2 — 用户体验问题 |
| **现象** | 输入无效设备序号时静默选择第一个设备，用户无法感知选择失败 |
| **修复方案** | 增加 `fmt.Println("无效选择，默认使用第一个设备:", devices[0])` 提示 |
