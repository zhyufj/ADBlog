# 项目开发总结报告

## 1. 项目概述

ADBlog 是一个 Windows 命令行工具，通过 ADB 实时查看 Android 设备日志。Go 语言开发，单文件编译，即插即用。项目源于实际测试工作中的痛点——不同品牌手机导出日志方式各异、操作繁琐。

### 3.2 关键技术问题与解决

#### 问题 1：extractLevel 消息体误判（P0 Bug）

**问题描述：** 日志消息体中的 "I/Stream"、"D/Config" 等文本被错误识别为日志级别标记。

**根因分析：** 旧版使用 `strings.Contains` 在整个日志行（前50字符）中搜索 " X/" 或 " X " 模式，无法区分该模式出现的位置是真正的级别标记位还是消息体中的偶然匹配。

**解决方案：** 改为逐字符扫描算法，增加前导字符验证（必须为空格或行首）和后继字符验证（必须为 `/` 或空格），确保只匹配语法位置正确的级别标记。

**效果：** 消除消息体误判问题，已编写 6 条专项单元测试验证。

#### 问题 2：多设备选择无反馈（P2 体验）

**问题描述：** 用户输入无效序号时静默选择第一个设备，无任何提示。

**解决方案：** 增加 `fmt.Println("无效选择，默认使用第一个设备:", devices[0])` 提示信息。

## 5. 项目亮点

1. **聚焦实际问题：** 源于真实测试工作中的痛点，解决测试人员获取日志繁琐的问题
2. **零学习成本：** 双击运行、交互式菜单引导，不需要记忆任何命令
3. **即插即用：** 单文件 EXE + 自带 adb.exe，解压即用，无需安装配置
4. **核心算法严谨：** extractLevel 逐字符扫描算法经过完整单元测试，消除了消息体误判的隐蔽 Bug
5. **文档体系完整：** 编写了 8 份软件工程文档 + 61 条功能测试用例 + 单元测试代码，覆盖开发全流程

## 6. 不足与改进

| 不足 | 改进方向 |
|------|----------|
| 仅支持 Windows | 通过 `runtime.GOOS` 判断平台，动态设置 adb 路径 |
| 包名匹配大小写敏感 | 使用 `strings.ToLower` 进行大小写不敏感匹配 |
| 不支持日志保存 | 增加 `> output.log` 重定向或 `-o` 参数导出文件 |
| ADB 命令无超时控制 | 为 `exec.Command` 设置 context 超时 |
| 无日志回滚/翻页 | 可考虑集成 less/more 风格的分页查看 |
| 单文件架构扩展性受限 | 若功能持续增加，适时拆分为多包结构 |

---

## 7. 经验教训

1. **单元测试要趁早：** extractLevel 的 Bug 如果在编码阶段就写好单元测试，可以第一时间发现，而非等到功能测试阶段
2. **边界条件容易遗漏：** 消息体文本与级别标记格式相同是小概率但高风险场景，设计阶段应做更全面的边界分析
3. **用户体验细节重要：** 多设备选择增加一行提示信息只需 1 行代码，但能显著改善用户体验
4. **文档驱动开发：** 先写需求文档再编码，能够帮助理清思路，减少返工
