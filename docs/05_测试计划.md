# 测试计划

## 1. 测试目标

- 验证全部 15 条功能需求均已正确实现
- 通过单元测试 + 功能测试 + 集成测试三层验证软件质量
- 为测试报告提供执行依据

## 2. 测试范围

### 2.1 测试对象

| 测试对象 | 说明 |
|----------|------|
| 源代码 | `main.go`（单一源文件，约290行） |
| 可执行程序 | `ADBlog.exe`（Go 编译产物） |
| 外部依赖 | `adb.exe`（程序自带） |

## 3. 测试策略

### 3.1 单元测试策略

| 维度 | 说明 |
|------|------|
| 测试框架 | Go `testing` 标准库 |
| 测试方法 | 表驱动测试（Table-Driven Tests） |
| 覆盖重点 | extractLevel 算法正确性（含 Bug 修复验证）、chooseLevel 输入校验、processLine 过滤逻辑 |
| 不覆盖 | 依赖 `exec.Command` 的函数（selectDevice、getCurrentPackage 等） |
| 通过标准 | 所有用例 PASS，核心函数（extractLevel / chooseLevel）无失败 |

### 3.2 功能测试策略

| 维度 | 说明 |
|------|------|
| 测试方法 | 黑盒测试、等价类划分、边界值分析、场景测试 |
| 测试环境 | 真机 + Windows 终端 |
| 用例设计 | 按10个功能模块逐一设计，覆盖正常流程、边界条件、异常场景 |
| 通过标准 | P0 用例 100% 通过，P1 用例 ≥ 95% 通过 |

### 3.3 集成测试策略

| 维度 | 说明 |
|------|------|
| 测试方法 | 端到端场景测试，覆盖3条核心用户路径 |
| 测试环境 | 真机 + 完整 ADB 链路 |
| 通过标准 | 3条路径全部通过，无阻塞性缺陷 |

## 7. 缺陷管理

### 7.1 缺陷严重等级

| 等级 | 定义 | 示例 |
|------|------|------|
| P0-致命 | 核心功能完全不可用 | 级别误判导致过滤结果错误 |
| P1-严重 | 功能可用但有明显错误 | 设备选择无提示、颜色显示错误 |
| P2-一般 | 边缘场景或体验问题 | 提示信息不够清晰 |
| P3-建议 | 优化建议 | 增加更多日志格式支持 |

### 7.2 缺陷生命周期

```
发现 → 提交 → 确认 → 修复 → 验证 → 关闭
                  ↑              ↓
                  └── 拒绝 ──────→ 关闭
```

### 7.3 缺陷跟踪工具

使用 Git Issues 或 Excel 表格跟踪管理。

## 8. 风险评估

| 风险 | 影响 | 应对 |
|------|------|------|
| 真机设备不足 | 兼容性测试不充分 | 至少准备 2 台不同品牌设备 |
| ADB 版本兼容 | 部分设备连接失败 | 使用最新的 adb.exe |
| 日志输出格式差异 | 级别提取失败 | 同时支持 brief 和 threadtime 格式 |
| 时间不足 | 功能测试压缩 | 优先保证 P0 用例执行 |
